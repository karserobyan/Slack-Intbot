## 3. The problem we're solving next

You reported: **"the bot is dumbed out" and "throws Something went wrong error a lot recently"**.

### Suspects ranked by likelihood

1. **`parseClaudeResponse` throws on malformed JSON** (`src/claude/prompts.js:103-120`) — the recent `HARD RULE — MANDATORY SEARCHES` (added in commit `cb8407f`) forces Claude to call Slack MCP before *any* JSON output. When Slack MCP fails, Claude sometimes emits prose explanation instead of JSON, and the parser throws. This is OUTSIDE the network try/catch in `query.js:146`, so it bubbles up to the "Something went wrong" branch.

2. **Slack MCP failures** — `buildMcpServers()` silently returns `[]` if `SLACK_USER_TOKEN` is missing/placeholder. Combined with the mandatory-search rule, Claude gets stuck.

3. **Atlassian REST 401** — if the token in your `.env` was generated by the now-deleted OAuth PKCE script, it won't authenticate with Basic Auth. `searchConfluence/searchJira` silently return `null`. Result: Claude gets no Confluence/Jira context → "dumbed out" answers.

4. **Verbatim CQL/JQL searches** — `atlassian-search.js:36, 82` use raw user text as `text ~ "..."` queries. Email pastes, redundant references, names in the question all become noise in the search.

5. **Prompt drift** — `prompts.js:191, 329` still say "use your atlassian and slack search tools" even though Atlassian is no longer an MCP tool. Claude can hallucinate an Atlassian tool call.

### Recent commits that introduced these regressions (timeline)

| Commit | What it did | Why it bites now |
|---|---|---|
| `7ef6cf4` (this branch) | Replaced Atlassian MCP with REST | Prompts still reference Atlassian as an MCP tool (drift). REST uses Basic Auth — old OAuth tokens fail silently. |
| `cb8407f` | Added "MANDATORY SEARCHES" hard rule | Claude MUST call Slack MCP before any JSON. If Slack MCP fails, Claude returns prose → parser throws. |
| `d378fee` | Removed knowledge fast-lookup shortcut | All queries now always hit MCP; no escape hatch when MCP is sick. |

### Diagnostic still owed

We agreed the redesign supersedes a targeted fix, but **a single failing-query log would confirm which suspect is firing**. When resuming, you can either:
- (a) skip diagnosis and trust the redesign to fix all five (faster, riskier)
- (b) grab one stderr capture of a failing query first to confirm root cause (slower, safer)

### Why the redesign also FIXES the regression (not just an enhancement)

This is the key insight that justifies the scope:

- The new pipeline **deletes the `MANDATORY SEARCHES` hard rule** — the planner decides which sources to hit, so no prompt-level enforcement is needed → suspect #1 vanishes.
- Slack moves from MCP to direct Web API → suspect #2 vanishes.
- The Interpreter generates targeted CQL/JQL keywords instead of verbatim text → suspect #4 vanishes.
- The new prompt files are written from scratch with REST-aware language → suspect #5 vanishes.
- Suspect #3 (Atlassian token type) is fixed by you re-issuing the API token from `id.atlassian.com/manage-profile/security/api-tokens`. That's an env fix, not a code fix.

So the redesign is **simultaneously a feature (better question understanding) and a refactor (eliminates the failure modes)**. That's why we agreed to do it instead of patching.

## 4. The redesign we're brainstorming

You want the bot to:
- **Break down the question** — strip email noise, names, redundant references
- **Understand what's really needed** — extract intent, integration, error code, tenant
- **Find the perfect way to find the answer** — pick the right sources, the right keywords, iterate if first pass missed

### Decisions made so far

| Question | Answer |
|---|---|
| Failure mode to fix | All three (vague, wrong-source, compound) + email/name noise + redundant references |
| Should bot confirm interpretation with user? | **Only when interpretation is uncertain** — high-confidence answers run silently; medium/low show interpretation and ask |
| Search strategy after interpretation | **Source-aware planning + iterative refine when needed** (combined option 2+3) |
| Latency budget | **Under 30s typical, 60s max** — limits agent loop to 2 rounds |

### Alternatives considered (rejected)

| Approach | Why rejected |
|---|---|
| **A. Interpreter → Search → Answerer** (linear, no evaluator) | Too rigid. No "if results don't match, search again" loop — hard questions with subtle keywords would still miss. Fast (always 2 model calls) but ceiling on quality is low. |
| **C. Single Claude call doing plan + search + answer** | Closest to today's code. Confidence gating is hard inside one streaming turn (can't easily early-return a clarifying question). Email noise still leaks into Claude's keyword choices because there's no separation of "understand" from "answer". |

### Approach selected: **B — Interpreter → Search → Evaluator → (Refine) → Answerer**

```
[raw question]
    ↓
[Haiku: clean & interpret]                ~2s
    ├─ low confidence → clarifying question. Done.
    └─ ok → { cleaned_question, intent, entities, search_plan }
    ↓
[Parallel search round 1]                 ~5s
    ↓
[Haiku: evaluator — sufficient?]          ~2s
    ├─ yes → skip refinement
    └─ no → emit a refined search_plan
                ↓
                [Parallel search round 2] ~5s
    ↓
[Sonnet: answer using cleaned question + all results]  ~15s
```

Total: ~25s fast path, ~35-45s when refinement fires. Hard cap: 1 refinement only.

### Architecture (section 1 of the design — ✅ user-approved 2026-05-13)

| Component | File | Model | Job |
|---|---|---|---|
| Interpreter | `src/claude/interpreter.js` (new) | Haiku 4.5 | Strip email noise, extract entities, judge confidence, build search plan |
| Search Executor | `src/claude/search-executor.js` (new) | none (direct API) | Run the plan in parallel: KB / Confluence / Jira / **Slack via Web API, not MCP** |
| Evaluator | `src/claude/evaluator.js` (new) | Haiku 4.5 | Judge if results match cleaned question; emit refined plan if not |
| Answerer | `src/claude/answerer.js` (refactored from `queryWithContext`) | Sonnet 4.6 | Produce final structured response |
| Orchestrator | `src/claude/pipeline.js` (new) | — | Coordinates the four stages |

**Other key changes the design implies:**
- **Slack moves from MCP to direct Web API** (`search.messages`). Removes the `HARD RULE — MANDATORY SEARCHES` failure mode entirely. Removes MCP plumbing from `query.js`.
- **Pre-fetch goes away.** Today `mention.js:163-174` always pre-fetches KB/Confluence/Jira with verbatim text. In the new design the planner picks sources + keywords.
- **`prompts.js` splits into three** files (interpreter / evaluator / answerer prompts).
- **The mandatory-search rule deletes itself** — the planner decides which sources to hit, so the prompt doesn't need to enforce a Slack call.

**Open decisions inside Section 2 — user must approve / push back:**

1. **`intent` enum:** `troubleshooting | how-to | policy | audit | unclear`. `audit` already exists as a separate Kibana flow; `unclear` triggers the clarifying-question path.
2. **`entities` schema:** minimal 5-field (integration, error_code, tenant_id, customer_mentioned, symptom). YAGNI on `job_type` / `error_message_text` for v1.
3. **`confidence` thresholds:**
   - `high` — integration named AND symptom clear AND no ambiguity
   - `medium` — one side named, the other vague (still searches; Answerer is told to lean toward follow-up if results aren't strong)

4. **`medium` behavior:** runs the search plan, but Answerer prompt is told "interpreter wasn't fully confident; if results don't strongly support a specific answer, output a `clarifying_question` instead of a full structured response". Lets the search rescue a fuzzy question — but lets the Answerer back out if it can't.

## 8. Reference: full request flow today (for orientation)

When a `@mention` or DM arrives, both go through `handleQuery` in `src/handlers/mention.js:49`:

1. Strip `<@U…>` mention
2. Empty query → greeting + return
3. Rate limit (5/min/user) → return if exceeded
4. Accounting fast-path (keyword match) → redirect, return
5. `help` command → help card, return
6. **Thread has history** → follow-up branch (`queryChat`): post "Thinking…", parallel pre-fetch KB/Confluence/Jira via REST, stream Claude with Slack MCP, return resolution or follow-up question
7. Cache hit → return cached response
8. Role detection (CSA/Specialist) + post "Checking…" in parallel
9. Inject sanitized feedback corrections
10. **Full Claude query** (`queryWithContext`): pre-fetch KB/Confluence/Jira, stream Claude with Slack MCP, `parseClaudeResponse` on the joined text
11. Attach metadata, cache result conditionally
12. Accounting double-check
13. Clarifying question early-return
14. Deliver response card
15. Seed conversation history
16. Nominate response for knowledge base (if high-quality + has refs)

The new pipeline replaces steps 6, 10 (and the pre-fetch logic) with the four-component flow described in Section 4 above.
